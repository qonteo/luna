api.luna.visionlabs.ru

Introduction

Welcome to Luna, a service that provides a comprehensive set of facial analysis algorithms.

The service solves several common tasks: face detection with attributes, grouping and recognition.

The API functions are divided into several blocks:

  • /account API - client authorization details and usage statistics
  • /storage API - functions to detect faces and store their descriptions within the system
  • /matching API - functions to perform person identification and verification.

Account API

Resource for accessing current account parameters. The account is identified by an authorization scheme in use (basic authorization or API token).

Data accessible this way:

  • account registration info
  • account usage statistics
  • list of API tokens.

Some APIs are secured with login + password authorization as they might give access to potentially sensitive data.

The others are accessible via tokens. Multiple tokens may be generated per account. A token might be used to authorize one or multiple clients. If a token is withdrawn, all clients authorized by the token will loose access to API until a valid token is provided.

See /account API section for details.

See /accounts API section for details on how to create new accounts.

Storage API

Resource for accessing current account data. The account is identified by an API token.

Data accessible this way:

  • person face descriptors
  • person data, including linked face descriptors and user data
  • person and descriptor lists
  • person portrait images.

Resource storage policies

Persons and lists are persistent resources; therefore, common CRUD access patterns are implemented for them.

Descriptors and their corresponding portraits, however are garbage-collected. This means that once created, a descriptor is put into storage. A TTL (time to live) is assigned to the descriptor. Default TTL is 24 hours. Note, that system administrators may choose to adjust this value or even disable garbage collection entirely. Once TTL is expired, the descriptor may be evicted to free up memory for new ones. Note, that TTL is reset to it's maximum value upon each descriptor access (e.g. matching).

To make a descriptor persistent, one should link it to a person or add it to a list. Descriptors linked this way will have infinite TTL, i.e. they will never be evicted. However, if a descriptor gets detached from all persons and lists it was previously attached to, it's TTL is reset back to default. If not used ever since, the descriptor will be eventually garbage-collected once it's TTL expires.

To link a descriptor, use /persons/*/linked_descriptors and /descriptors/*/linked_lists API.

Person portraits are associated with the corresponding descriptors and obey the same rules.

Due to GC logic, there is no DELETE request for /descriptors and /portraits resources. As a result of using garbage collection, it is possible to maintain high performance with only really relevant data being kept almost entirely in RAM.

See /storage API section for further details.

Matching API

This resource provides access to face matching workers. The workers are capable of /identifying a person from a given person list by face and /verifying a person by face. Person data is taken from the current account /storage. The account is identified by an API token.

There is also a simplified /search sub resource that combines face detection followed by identification into a single request.

The /match sub resource may be used to search similarities for both descriptors that are attached and not attached to persons (called 'plain descriptors' later on).

See /matching API section for details.

/storage

Storage API provides access to all persistent data for the current account.

/storage/persons post get

Persons API allows to create and manage persons. Each person is identified by a unique id and stores the following:

  • an array of descriptors linked to this person;
  • an array of lists this person is linked to;
  • a user-defined data (optionally).

To be able to verity or identify a person, one should attach at least one face descriptor to it. Generally, multiple face descriptors may be attached, but it's not required. Multiple descriptors per person may sometimes improve matching results.

Additionally, some user defined data may be attached to a person. Luna does not use this data and has no way to interpret it. The data is stored as a short string and may be used to aid integration with other systems.

User data size is limited to 128 characters for performance reasons.

/storage/descriptors post get

Descriptors API allows to create and manage human face descriptors from photographic images. A descriptor contains facial features stored in a compact way. Note, that it is impossible to convert a descriptor back to it's source image.

All extracted descriptors are stored in the current account database namespace. For visualization purposes Luna also stores a portrait image for each extracted descriptor; these images are represented by /portraits resource.

Once a descriptor is extracted from a face, it can be used to search for similar faces by their descriptors (see matching API for details on searching).

post

Create descriptor(s) from an image.

In order to extract any descriptors, the function performs face detection first. In case of multiple detected faces, the function will produce several descriptors (one per face).

Please note that some faces may not be detected for technical challenges, e.g. very large face angles (head-pose) or large occlusion. Frontal and near-frontal faces have the best results.

The detectable face size is between 96x96 to 1024x1024 pixels. The faces out of this range will not be detected. A maximum of 64 faces could be returned for an image. The returned faces are ranked by face detection confidence in descending order.

Input image size should be at least 96x96 pixels and no more than 4096x4096 pixels, images not meeting the criteria will be rejected with HTTP status code 400.

Supported image formats and their content-types:

Image formatContent-Type
Jpegimage/jpeg
Pngimage/png
Bmpimage/bmp
Tiffimage/tiff
Gifimage/gif
PPMimage/x-portable-pixmap

This function always outputs detected faces positions as bounding rectangles. Each rectangle is defined by it's top-left corner position, width and height. Note, that coordinate system origin is at the top-left corner of the image. The face rectangles are stored in rect objects per face.

Aside from face detector response, the function also outputs bounding rectangles scaled and biased to resemble conformance to ISO IEC 19794-5 2005(E).

To 'resemble conformance' here means that the standard dictates not only the bounding rectangle shape but also photoshooting conditions. Since Luna is a general system and not a true biometric one, it has no means to guarantee the conditions. However, it can detect person's eyes on a picture and build a rectangle with proportions described in the standard. The best results are observed on non-rotated faces while pose variations introduce errors.

Note, that the procedure of ISO rectangle construction does not take image boundaries into account. The resulting rectangle may outstep the image boundaries. The client is responsible to do proper padding if image cropping is desired.

The ISO face rectangles are stored in rectISO objects per face.

While face detection step is mandatory, all the remaining steps can be enabled or disabled. E.g. while descriptor extraction is enabled by default, the client may disable it provided only face bounding boxes are required. See additional feature descriptions in the query parameters section below.

In order to make face descriptor invariant to some pose variations and scale, each face detection area is cropped from the source image. Then several affine transformations are applied to the crop including rotation and scaling. This process is called warping. Since warping scales all cropped images to the same size, one won't get a much better face descriptor from a higher resolution photograph.

In other way around, while it is possible to extract descriptors from smaller images, matching quality of such descriptors will be lower due to the lack of detail. To give an idea, reasonable face bounding box size is about 150x150 pixels.

It is possible to estimate additional per face attributes, including:

  • overall face image quality (suitablity for recognition);
  • age;
  • gender;
  • glasses;
  • ethnicities;
  • emotions.

These attributes are not stored in the DB.

/matching

Matching API provides access to various searching services.

/matching/identify post

Identification is done via matching a reference descriptor with a list of candidate persons' descriptors.

The reference descriptor may be specified explicitly or taken from a person defined by the person id.

The matching process produces similarities for each candidate. Maximum of top-5 most similar results are then selected. Identification result is therefore a short list of the most similar persons and their similarity scores. It's up to user to decide whether the similarity score high enough to consider it a successful match or not.

post

Match a descriptor or person to a list of candidate persons.

Only one of descriptor_id or person_id parameters should be specified as the reference at a time. You can not set both at once. Also only one of list_id or person_ids parameters should be specified as the candidate.

HTTP status code 200

OK

Headers

  • Content-Type: required (application/json)

    An indicator of the media type of the response body.

  • LUNA-Request-Id: required (string)

    Request id. Helps uniquely identifying messages, corresponding to particular requests, in system logs.

    Example:

    1516179740,c06887a2

Body

Media type: application/json

Type: object

Properties
  • Version: required (object)
    • luna_api: required (object)

      Luna API versions

      • api: required (integer - minimum: 1)

        api version number.

      • major: required (integer - minimum: 0)

        Major version number.

      • minor: required (integer - minimum: 0)

        Minor version number.

      • patch: required (integer - minimum: 0)

        Patch or revision number.

    • luna_core: required (object)

      Server and SDK versions are reported as structures with three member

      • api: required (string)

        api version number.

      • fsdk: required (object)

        Underlying Luna SDK (FaceEngine) version

        • major: required (integer - minimum: 0)

          Major version number.

        • minor: required (integer - minimum: 0)

          Minor version number.

        • patch: required (integer - minimum: 0)

          Patch or revision number.

      • luna: required (object)

        Luna server version

        • major: required (integer - minimum: 0)

          Major version number.

        • minor: required (integer - minimum: 0)

          Minor version number.

        • patch: required (integer - minimum: 0)

          Patch or revision number.

Example:

{
  "Version": {
    "luna_api": {
      "patch": 13,
      "minor": 1,
      "major": 0,
      "api": 1
    },
    "luna_core": {
      "api": "10",
      "fsdk": {
        "major": 2,
        "minor": 8,
        "patch": 3
      },
      "luna": {
        "major": 2,
        "minor": 0,
        "patch": 7
      }
    }
  }
}
